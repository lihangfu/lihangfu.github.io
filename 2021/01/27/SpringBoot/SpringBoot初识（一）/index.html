基于 Java 的配置

为什么要在这里提纯 Java 的配置，因为在 SpringBoot 中的自动配置原理，很多便是通过 Java 配置，去重写一些方法

  • 配置 Spring

在 SpringBoot 中便是通过重写父类的各种方法配合条件注解完成自动化配置，所以自动化配置并不是 SpringBoot 所特有的新功能

SpringBoot 介绍

  • 可以创建独立运行 Spring 应用
  • 内嵌 Tomcat，可以不用在打包成 war 包
  • 自动化配置，简化配置
  • 提供自动配置 Spring 和很多的第三方库的配置
  • 提供健康检查，监控程序
  • 没用代码生成，无需 xml 配置

@Configuration 详解

类注解，声明该类为配置类

我们可以看到，@Configuration 注解定义上继承了 @Component ，说明配置类也是注入到 Spring 中的一个组件，当我们使用 @Component 替换 @Configuration 注解时，发现功能一样可以实现，那么二者的区别在哪？

在 Spring 中有很多的后置处理器（PostProcessor），其中有一个 ConfigurationClassPostProcessor 专门处理带有 @Configuration 注解的类，主要有方法 enhanceConfigurationClasses 。

该方法在第一个 for 循环中找到所有带有 @Configuration 注解的类，并保存起来。

@SpringBootApplication 详解

组合注解，主要有以下几个注解：

  • @SpringBootConfiguration ：就是继承了 @Configuration 注解
  • @EnableAutoConfiguration ：开启自动化配置
  • @ComponentScan ：包扫描，默认扫描当前类目录及子目录下注解

@SpringBootConfiguration 本质上就是 @Configuration 我们不再细说，主要看看下面两个注解。

@Import 详解

@Import 时 Spring 原生注解

在原生的 Spring 中，组件装配有三个阶段：

  1. Spring 2.5+ ，@Component
  2. Spring 3.0+ ，使用 @Configuration + @Bean
  3. Spring 3.1+ ，使用 @EnableXXX + @Import

前两种普遍在组件较少的情况，后一种多用于框架组件较多的情况

在 SpringBoot 中便是大量采取第三种方式，我们来学习一下，如何使用第三种方式完成组件装配：

@EnableAutoConfiguration

继承注解：

  • @AutoConfigurationPackage 继承 @Import({Registrar.class})
  • @Import({AutoConfigurationImportSelector.class})

所以这两个注解的实际作用，就是导入这两个配置类 Registrar 和 AutoConfigurationImportSelector

二者都是用于加载自动化配置

@ComponentScan

注解扫描，没有指定扫描包路径，默认扫描当前类路径下所有的注解。

使用 excludeFilters 排除一些注解不扫描，主要排除自动化配置类，在上个注解已经扫描，避免重复扫描
